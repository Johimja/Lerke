# Lerio Backend Model

## Goal

Define a shared backend model for Lerio so multiple learning activities can use the same core platform.

This is a product and architecture draft, not a final database schema.

The purpose is to avoid building Bingo as a one-off backend and then having to rebuild everything when Lerio grows.

---

## Core Idea

Lerio should have:

- a shared backend foundation
- shared session and participant logic
- activity-specific state layered on top

That means:

- Bingo should use the same session model as future tools
- a future writing app should not need a completely separate backend architecture
- teacher, student, group, and session concepts should be reusable across activities

---

## Recommended Platform

Recommended backend platform:

- Supabase

Why:

- PostgreSQL is well suited to structured school/activity data
- row-level security is useful for teacher/student permissions
- realtime supports live classroom sessions
- auth can be added gradually
- storage can support exports, uploads, and future assets

---

## Shared Building Blocks

These are the core concepts Lerio should reuse across activities.

### `activities`

Defines what kind of tool is being used.

Examples:

- bingo
- quiz
- writing
- wordchain

Possible fields:

- `id`
- `slug`
- `name`
- `status`
- `created_at`

---

### `sessions`

A live or temporary classroom session for one activity.

This is the main unit students join.

Examples:

- Bingo session for 7A
- Quiz session for chapter 5
- Writing workshop for group 2

Possible fields:

- `id`
- `activity_id`
- `join_code`
- `title`
- `created_by`
- `status`
- `settings`
- `expires_at`
- `created_at`

Notes:

- `join_code` should be short and classroom-friendly
- `id` should still be a strong internal identifier
- `settings` can hold activity-specific config if needed

---

### `participants`

Represents people or devices in a session.

Examples:

- teacher
- student
- anonymous student device

Possible fields:

- `id`
- `session_id`
- `user_id` nullable
- `role`
- `display_name`
- `group_id` nullable
- `joined_at`
- `last_seen_at`
- `status`

Typical roles:

- teacher
- student
- observer

For student-facing V1 flows:

- `display_name` should be generated by the system
- no free-text nickname entry
- allow a small number of rerolls during join

---

### `groups`

Optional grouping inside a session.

This is useful for:

- team-based quiz
- collaborative writing
- pair/group activities

Possible fields:

- `id`
- `session_id`
- `name`
- `color`
- `created_at`

Not every activity needs groups, but the model should support them.

---

### `session_state`

Stores the current live state of a session.

This is what most clients subscribe to in realtime.

Examples:

- current round in Bingo
- current question in Quiz
- current phase in Writing

Possible fields:

- `session_id`
- `phase`
- `round_number`
- `step_index`
- `current_payload`
- `updated_at`

Examples of `current_payload`:

- current draw in Bingo
- active question in Quiz
- active writing prompt in Writing

---

### `session_events`

Append-only event log for what happens during a session.

Examples:

- session_started
- round_started
- draw_advanced
- bingo_reported
- answer_submitted
- text_updated

Possible fields:

- `id`
- `session_id`
- `participant_id` nullable
- `event_type`
- `payload`
- `created_at`

Why this matters:

- draw history
- replay/debugging
- auditability
- future analytics

---

### `participant_state`

Stores per-participant live or accumulated state.

Examples:

- bingo status
- quiz score
- writing word count
- activity progress

Possible fields:

- `id`
- `session_id`
- `participant_id`
- `state_type`
- `state_payload`
- `updated_at`

This should be flexible enough to support different activities.

---

### `artifacts`

Stores the content or materials belonging to an activity.

Examples:

- bingo word list
- generated round sequence
- quiz question set
- writing document
- group text draft

Possible fields:

- `id`
- `session_id`
- `artifact_type`
- `owner_type`
- `owner_id`
- `content`
- `created_at`
- `updated_at`

This becomes especially useful for writing and quiz features.

## How Bingo Fits This Model

### Shared tables used by Bingo

- `activities`
- `sessions`
- `participants`
- `session_state`
- `session_events`
- `participant_state`
- `artifacts`

- title = "Kapittel 5 · 7A"

#### Session state

- phase = live_draw
- round_number = 2

- draw_advanced
- round_reset
- session_finished
- bingo_reported

#### Participant state

- marked count
- bingo reported

- join_code = classroom code

### Session state

- phase = brainstorm / write / revise

### Participants

- individual students
- teacher

### Participant state

- word count
- completion %

- contribution amount

### Artifacts

- each student text
- each group text
- shared class text
- prompt content

### Session events

- text_updated
- phase_changed
- student_finished
- teacher_feedback_posted

This model works for:

- individual writing
- pair writing
- group writing
- teacher-led progression tracking

## Security Model

The shared backend should follow the same principles across all activities.

### Teacher permissions

Teacher can:

- create sessions
- control session flow
- view participant states
- manage groups
- publish prompts/questions/draws

Recommended identity model for V1:

- teacher has an individual account
- teacher role is not automatic on signup
- teacher access requires approval or invite-key activation

### Student permissions

Student can:

- join sessions
- read allowed session state
- update only their own participant state
- create/update only their own allowed artifacts

### Hard rules

- role is never trusted from the client alone
- teacher actions require server-side verification
- students must not write to global session control fields
- activity permissions should be enforced at row level

---

## Design Principle: Generic Core, Specific Payloads

The best model for Lerio is:

- generic shared tables
- activity-specific payload content

That means:

- the session system stays stable
- the data inside `current_payload`, `state_payload`, and `content` can vary by activity

This prevents overfitting the whole backend to Bingo only.

---

## Suggested V1 Scope For Shared Backend

The first production backend should still stay small.

Build shared infrastructure for:

- sessions
- participants
- session_state
- session_events

Then let Bingo be the first activity that uses it.

Do not try to fully generalize everything up front.

The right approach is:

1. design shared concepts now
2. implement them through Bingo first
3. extend carefully when Quiz/Writing arrives

---

## Practical Recommendation

For Lerio as a platform:

- use Supabase as the shared backend
- design core tables around sessions, participants, state, events, and artifacts
- let Bingo be the first implementation
- keep future writing and quiz support in mind, but do not overbuild before needed

---

## Concrete Next Steps

1. Accept this shared model as the baseline direction.
2. Translate Bingo V1 into concrete Supabase tables and policies.
3. Decide which shared tables are needed immediately and which can wait.
4. Build Bingo live sync as the first real backend feature.
5. Reuse the same model later when the first writing prototype is designed.
