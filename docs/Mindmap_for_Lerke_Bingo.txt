Lerke Bingo: Strict Digital Live Gameplay Spec

 1. Design goal

 Lerke Bingo should behave as a teacher-driven, live, authoritative classroom game, not as a locally self-managed bingo
 sheet.

 That means:

 - teacher controls the pace
 - students respond to the current live draw only
 - answers are time-limited
 - students cannot retroactively answer missed draws
 - local actions cannot invalidate the live game state

 ────────────────────────────────────────────────────────────────────────────────

 2. Core principles

 ### 2.1 Teacher is authoritative

 The teacher session defines:
 - what the current prompt is
 - when answering is open
 - when answering is closed
 - when a round starts/ends
 - when cards reset
 - when a session ends

 ### 2.2 Student board is persistent

 Each student gets a generated board tied to:
 - session
 - participant
 - round/card state

 It is not just a temporary local board.

 ### 2.3 One draw = one answer window

 For every teacher draw:
 - there is one active prompt
 - students may answer only during that draw window
 - once closed, that draw cannot be answered anymore

 ### 2.4 Only correct answers permanently mark the board

 A correct response:
 - permanently marks the correct square on the student board

 An incorrect response:
 - does not permanently mark the square
 - but consumes the student’s chance for that draw

 ────────────────────────────────────────────────────────────────────────────────

 3. Session structure

 3.1 States

 A Bingo session should move through these states:

 ### lobby

 - teacher has created session
 - students may join
 - cards may be generated/assigned
 - no active draw yet

 ### round_active

 - a round is running
 - prompts are being revealed one by one

 ### draw_open

 - current prompt is active
 - students may answer

 ### draw_locked

 - current prompt is no longer answerable
 - teacher may review briefly or move on

 ### round_complete

 - all prompts for the round are done, or teacher ends round
 - winners can be shown
 - teacher may reset for next round

 ### session_complete

 - whole session is done

 ────────────────────────────────────────────────────────────────────────────────

 4. Teacher flow

 4.1 Pre-game setup

 Teacher chooses:
 - game type:
     - Glosebingo
     - Mattebingo
 - content set
 - session name
 - number of rounds
 - answer window duration
     - e.g. 3, 5, or 8 seconds
 - optional rules:
     - allow retry within same draw? default no
     - reveal correct answer after lock? default yes
     - auto-advance next draw? default teacher manual

 Teacher then:
 - creates session
 - opens lobby
 - students join
 - teacher starts round

 ────────────────────────────────────────────────────────────────────────────────

 5. Student flow

 5.1 Join

 Student joins via:
 - QR code
 - join code
 - optionally with student account already active

 On join, student gets:
 - participant identity
 - generated nickname or account-linked display name
 - assigned bingo card for the session/round

 5.2 Before round starts

 Student sees:
 - session title
 - own nickname/name
 - card
 - waiting state:
     - “Venter på at læreren starter”

 No cell can be selected yet.

 ────────────────────────────────────────────────────────────────────────────────

 6. Board generation rules

 6.1 One board per round or one board per session

 This must be a configurable product rule.

 ### Option A — same board reused across round

 - same board for whole session
 - teacher clears progress between rounds if needed

 ### Option B — new board each round

 - best for stricter live play
 - simpler logic
 - each round = fresh board

 Recommended:
 Use new board each round in strict live mode.

 That reduces confusion and avoids needing a student-side Tøm button.

 ────────────────────────────────────────────────────────────────────────────────

 7. Draw lifecycle

 7.1 Teacher opens a draw

 Teacher reveals a prompt.

 Examples:

 ### Glose

 - prompt shown: because
 - expected answer on student board: fordi

 ### Matte mode 1

 - prompt shown: 3 × 7
 - expected answer on board: 21

 ### Matte mode 2

 - prompt shown: 21
 - expected answer on board: 3 × 7

 When teacher opens draw:
 - session state becomes draw_open
 - draw index increments
 - timer starts

 7.2 Student answering

 During draw_open:
 - student may tap exactly one square
 - system checks whether tapped square matches expected answer for current draw

 Possible outcomes:

 ### Correct

 - selected square turns green
 - square becomes permanently marked
 - student gets feedback:
     - “Riktig!”
 - student cannot answer again on this draw

 ### Incorrect

 - selected square flashes red
 - feedback:
     - “Feil”
 - no permanent mark added
 - student cannot answer again on this draw

 ### No answer before timeout

 - feedback:
     - “Tiden er ute”
 - no mark added

 ────────────────────────────────────────────────────────────────────────────────

 8. Anti-retroactive rule

 Once a draw locks:
 - that prompt is over
 - student cannot answer it later
 - student cannot tap an old correct square and get credit

 This is one of the most important rules in the strict digital model.

 ────────────────────────────────────────────────────────────────────────────────

 9. Feedback model

 9.1 Immediate per-draw feedback

 Each student should see one of:

 - Riktig!
 - Feil
 - Tiden er ute

 9.2 Visual behavior

 Recommended:

 ### Correct

 - green fill
 - checkmark or glow
 - permanent

 ### Incorrect

 - red flash for ~600–1000 ms
 - then returns to normal appearance
 - no permanent damage to board

 ### Timed out

 - no color change to board
 - maybe small banner/text notice

 ────────────────────────────────────────────────────────────────────────────────

 10. Why wrong answers should not permanently burn cells

 You suggested red/black blocking. That is possible, but I recommend not doing that first.

 ### Recommended rule

 - wrong answer = lose this draw only

 ### Why

 - easier for younger students
 - less punitive
 - simpler to understand
 - still prevents cheating because they cannot retry after lock

 ### Optional future hard mode

 Teacher setting:
 - Feil svar blokkerer ruten
 default: off

 ────────────────────────────────────────────────────────────────────────────────

 11. Draw timer rules

 11.1 Timer ownership

 Timer is defined by teacher/session settings.

 Example:
 - 5 seconds to answer each draw

 11.2 Student view

 Student sees:
 - progress bar
 or
 - countdown ring / number

 11.3 Teacher control

 Two options:

 ### Manual advance

 - teacher opens draw
 - timer runs
 - draw locks
 - teacher presses next

 ### Auto advance

 - draw opens
 - timer runs
 - draw locks
 - after short reveal pause, next draw auto starts

 Recommended first version:
 Manual advance after lock.

 ────────────────────────────────────────────────────────────────────────────────

 12. Bingo detection

 12.1 What counts

 Standard:
 - 5 in a row
 - horizontal
 - vertical
 - diagonal

 12.2 Detection timing

 Bingo should be checked:
 - immediately after a correct mark is added

 If bingo is reached:
 - show student celebration
 - mark student as has_bingo
 - optionally notify teacher dashboard

 ────────────────────────────────────────────────────────────────────────────────

 13. Round completion rules

 A round ends when one of these happens:

 ### Option A

 Teacher finishes all draws

 ### Option B

 Teacher manually ends round

 ### Option C

 Teacher ends early after one or more Bingos

 Recommended:
 - teacher chooses when round ends

 ────────────────────────────────────────────────────────────────────────────────

 14. Clearing / resetting rules

 14.1 Student Tøm kort

 In strict live mode:
 - remove this during active session
 - or disable entirely

 Because it breaks authoritative play.

 14.2 Student Nytt kort

 In strict live mode:
 - remove or disable during joined session
 - only teacher/session should issue new board

 14.3 Teacher reset

 Teacher controls:
 - next round
 - new board
 - reset round
 - end session

 ────────────────────────────────────────────────────────────────────────────────

 15. Student UI spec

 Student UI should show:

 ### Header

 - session name
 - current round
 - student name/nickname

 ### Board

 - persistent card for current round

 ### Current live prompt area

 - current prompt
 - answer timer
 - current status:
     - waiting
     - answer now
     - locked
     - right
     - wrong
     - timed out

 ### Progress area

 - number of correct marks
 - BINGO count if relevant
 - wins/rounds if using multi-round session

 ### No local disruptive controls

 During strict live play:
 - no free Tøm
 - no free Nytt kort

 ────────────────────────────────────────────────────────────────────────────────

 16. Teacher UI spec

 Teacher should have:

 ### Lobby

 - participant count
 - participant names
 - start round

 ### Live control panel

 - current prompt
 - draw number
 - timer
 - open/lock state
 - next prompt
 - end round

 ### Teacher feedback

 Optional:
 - how many answered
 - how many correct
 - who has BINGO
 - near-BINGO indicators

 ────────────────────────────────────────────────────────────────────────────────

 18. Gameplay rules summary

 Recommended default rules

 - teacher-controlled live draw
 - one answer per draw
 - answer only while draw is open
 - no retroactive answering
 - correct = permanent green mark
 - wrong = red flash, no permanent mark
 - timeout = no mark
 - bingo checked after every correct mark
 - no student Tøm
 - no student Nytt kort
 - new board each round
 - teacher controls round reset

 ────────────────────────────────────────────────────────────────────────────────

 19. Suggested rollout plan

 Phase 1 — Safer live mode

 - disable Tøm
 - disable Nytt kort
 - show current live prompt on student side
 - show draw timer

 Phase 2 — Locked answer windows

 - only one answer per draw
 - no retroactive marking
 - draw open / locked states

 Phase 3 — Correctness feedback

 - green correct
 - red wrong
 - timeout feedback

 Phase 4 — Teacher overview

 - see who answered
 - see who has BINGO
 - progress dashboard

 20. Product decision points still needed

 You should decide these explicitly:

 1. One board per round or one board per whole session?
     - recommended: one board per round
 2. Should wrong answers permanently block cells?
     - recommended: no, not by default
 3. Should draws auto-advance or teacher manually advance?
     - recommended: manual first
 4. Should student account join and anonymous join behave the same in gameplay?
     - recommended: yes in gameplay, different only in identity/persistence
